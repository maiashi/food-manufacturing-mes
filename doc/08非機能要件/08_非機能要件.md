# 食品製造業MES - 非機能要件定義書

## 1. 本ドキュメントの目的

複数工場・複数製造ラインを擁する食品製造企業のMESシステムにおける、機能的でない要件（性能・可用性・セキュリティ・スケーラビリティ等）を定義する。既存システムとのIF連携も考慮する。

---

## 2. システム規模前提

| 項目 | 値 |
|------|-----|
| 工場数 | 3〜10工場（将来的な拡張を含む） |
| 製造ライン/工場 | 4〜12ライン（カテゴリ別に専用ラインあり） |
| ユーザー数 | 500〜2,000ユーザー（複数工場合計） |
| 1日当たり生産バッチ数 | 200〜800バッチ |
| 1日当たり在庫LOT数（原材料+完成品） | 1,000〜5,000 LOT |
| IoTセンサー接続数 | 500〜2,000デバイス（温度・重量・圧力等） |
| 月間API呼び出し数（IF連携含む） | 50万〜200万件 |
| データ蓄積量（1年） | 500GB〜2TB |

## 3. パフォーマンス要件

### 3.1 レスポンスタイム

| ID | 項目 | 要求事項 | 計測条件 |
|----|------|---------|---------|
| PF-001 | 画面遷移（通常） | ≤2秒（P95）、≤5秒（P99） | 全工場合計の最大ユーザー数時 |
| PF-002 | 検索・一覧表示（10万行以下） | ≤3秒（P95） | 複合条件フィルタ+ソート付き |
| PF-003 | 検索・一覧表示（10万〜100万行） | ≤10秒（P95） | ページネーション適用時 |
| PF-004 | CCP記録入力 | ≤1秒（即時フィードバック） | バッチIDスキャン→自動補完→保存まで |
| PF-005 | FIFO在庫割当て計算 | ≤5秒 | 最大2,000LOT同時引き当て時 |
| PF-006 | トレース検索（バック/フォワード） | ≤5秒（P95） | LOT ID指定、全工場横断検索 |
| PF-007 | バッチレポート生成 | ≤10秒（通常）、≤30秒（集計範囲 wide） | CSV/PDFエクスポート |
| PF-008 | IoTセンサーデータ受信処理 | ≤500ms/イベント | 2,000デバイス同時接続時 |
| PF-009 | API連携応答 | ≤2秒（通常）、≤10秒（バッチ一括） | 外部ERPへの同期API |

### 3.2 スループット

| ID | 項目 | 要求事項 |
|----|------|---------|
| SP-001 | 同時接続ユーザー数 | ≥500セッション同時保持 |
| SP-002 | 1秒当たりトランザクション | ≥200TPS（通常時）、≥500TPS（ピーク: おせち生産期間） |
| SP-003 | IoTデータ処理スループット | ≥1,000イベント/秒 |
| SP-004 | API連携処理能力 | ≥100要求/秒（外部システム向け） |
| SP-005 | バッチレポート並列生成 | 最大5件同時生成可能 |

### 3.3 データ処理性能

| ID | 項目 | 要求事項 |
|----|------|---------|
| DP-001 | データベース応答時間（単一検索） | ≤100ms（主キー・インデックス付き） |
| DP-002 | データベース応答時間（結合クエリ） | ≤500ms（3テーブルJOIN以下） |
| DP-003 | 一括INSERT性能 | ≥5,000行/秒 |
| DP-004 | バックアップ取得時間 | 日次フルバックアップ ≤1時間（2TB基準） |
| DP-005 | 月次集計ジョブ実行時間 | ≤30分（全工場データ） |

## 4. 可用性・信頼性要件

### 4.1 システム稼働率

| ID | 項目 | 要求事項 |
|----|------|---------|
| AV-001 | システム稼働率 | **99.95%以上**（月間障害時間 ≤ 22分）<br>※おせち生産期間(12月)は99.99%目標 |
| AV-002 | 計画停止 | 週次メンテナンス: 毎月第3火曜日 03:00〜05:00（2時間/月）<br>年次保守: 年2回、各4時間（夏季/冬季大規模メンテ） |
| AV-003 | 復旧目標時間(RTO) | **15分以内**（自動フェイルオーバー時）、4時間以内（手動対応時） |
| AV-004 | 復旧目標時点(RPO) | **5分以内のデータ損失**（継続的レプリケーション時） |
| AV-005 | ディザスタリカバリ(DR) | 地理的に離れたDRサイト: RTO≤2時間、RPO≤15分 |

### 4.2 バックアップ・障害対策

| ID | 項目 | 要求事項 |
|----|------|---------|
| BK-001 | データベースバックアップ | 継続的WALアーカイving + 日次スナップショット |
| BK-002 | ファイルバックアップ | 全ファイルストレージ: 毎時差分、30日間保持 |
| BK-003 | バックアップ検証 | 月1回DRサイトでの復旧テスト実施 |
| BK-004 | ハードウェア冗長化 | クラスタ構成（ノード≥3）。1ノード障害でサービス継続 |
| BK-005 | ネットワーク冗長化 | 2ISP契約 + BGP自動フェイルオーバー |

### 4.3 監視・アラート

| ID | 項目 | 要求事項 |
|----|------|---------|
| MN-001 | システム監視 | CPU/メモリ/ディスク/I/O/APIO/C/エラー率の24時間365日監視 |
| MN-002 | HACCPデータ監視 | CCP記録入力漏れをリアルタイム検知（30分以内通知） |
| MN-003 | IoTセンサー監視 | センサー接続切れ・データ異常値を検知・アラート |
| MN-004 | IF連携監視 | ERP等外部システム連携の成否をリアルタイム監視 |
| MN-005 | アラート通知経路 | メール + SMS + チャット(Slack/Teams)の3重経路 |

## 5. セキュリティ要件

### 5.1 アクセス制御

| ID | 項目 | 要求事項 |
|----|------|---------|
| SC-001 | 認証方式 | SAML 2.0 / OIDC対応。既存AD/LDAPとの連携可能 |
| SC-002 | MFA（多要素認証） | 管理者権限: 必須。一般ユーザー: オプション可 |
| SC-003 | セッション管理 | 30分自動タイムアウト、最大8時間 |
| SC-004 | 役割ベースアクセス制御(RBAC) | 工場/ライン/部署単位で権限を細粒度に設定可能 |
| SC-005 | データレベルアクセス制御 | 自工場データのみ表示（他工場データはアクセス不可） |

### 5.2 データ保護

| ID | 項目 | 要求事項 |
|----|------|---------|
| DP-010 | 通信暗号化 | TLS 1.3全リクエスト強制。内部通信もmTLS推奨 |
| DP-011 | 保存データ暗号化 | AES-256。HACCP記録は特に厳格管理 |
| DP-012 | データ改ざん防止 | 電子署名 + タイムスタンプ。HACCP記録の不可変性を保証 |
| DP-013 | PII保護 | 従業員情報等の個人情報を暗号化保存、アクセスログ付き |

### 5.3 監査証跡

| ID | 項目 | 要求事項 |
|----|------|---------|
| AU-001 | 操作監査ログ | 全操作をWho/When/Where(IP)/What/Whyを記録 |
| AU-002 | HACCP記録改ざん検知 | ハッシュチェーン方式で改ざんを検知 |
| AU-003 | 監査ログ保持期間 | 5年（HACCP関連は10年） |
| AU-004 | ログの整合性検証 | 月次SHA-256ハッシュによる整合性チェック |

## 6. IF連携要件（既存システムとの統合）

### 6.1 連携対象システム一覧

| ID | システム名 | タイプ | 連携方向 | 連携内容 | 頻度 |
|----|----------|--------|---------|---------|------|
| IF-001 | ERPシステム | SAP/Oracle等 | 양방향 | 受注情報取得、生産実績送信、在庫同期 | リアルタイム+バッチ(5分毎) |
| IF-002 | SCM/物流管理システム | WMS/TMS連携 | 양방향 | 出荷指示送信、入庫通知受信、配送情報連携 | リアルタイム |
| IF-003 | IoTセンサーゲートウェイ | MQTT/OPC-UA | 一方向(上り) | 温度・重量・圧力等センサデータ取得 | リアルタイム |
| IF-004 | ラベルプリンターサーバー | IPP/WiFi | MES→外部 | ロットラベル・商品ラベル印刷指示 | イベント駆動 |
| IF-005 | 品質分析ラボシステム(LIMS) | API連携 | 양방향 | COA取得、検査結果送信 | バッチ(1時間毎) |
| IF-006 | 会計システム | ファイル/API | MES→外部 | 生産原価データ送付 | 日次バッチ |
| IF-007 | HR/勤怠システム | API連携 | 一方向(下り) | 従業員マスタ同期、出勤情報取得 | 日次+リアルタイム |

### 6.2 インターフェース設計方針

| ID | 項目 | 要求事項 |
|----|------|---------|
| IF-101 | API設計 | RESTful API（OpenAPI 3.0準拠） + GraphQL（複雑な集計クエリ用） |
| IF-102 | メッセージング | Kafka/RabbitMQによるイベント駆動アーキテクチャ |
| IF-103 | データ変換 | マスタデータのマッピング定義をUIで管理可能にする |
| IF-104 | エラー処理 | 連携失敗時は自動リトライ（最大5回、指数バックオフ）、キューイング |
| IF-105 | 監視ダッシュボード | 全IF連携のステータスを一元可視化するダッシュボードを提供 |
| IF-106 | バッチ処理 | CSV/XMLファイルエクスポート/インポートにも対応（ERP等との互換性確保） |
| IF-107 | APIバージョン管理 | v1/v2並行稼働可能。廃止APIは6ヶ月以上の移行期間を設ける |

### 6.3 連携プロトコル一覧

| システム | プロトコル | データ形式 | ポート/エンドポイント |
|---------|-----------|----------|-------------------|
| ERP | REST API (HTTPS) / SFTP | JSON / EDI(X12, EDIFACT) | HTTPS:443 / SFTP:22 |
| WMS/TMS | REST API (HTTPS) | JSON | HTTPS:443 |
| IoT Gateway | MQTT over TLS | MQTT payloads | MQTT:8883(TLS) |
| LIMS | SOAP/REST API | JSON/XML | HTTPS:443 |
| HR System | OIDC + REST API | JSON | HTTPS:443 |
| プリンター | IPP / WiFi Direct | PDF/ZPL | IPP:631 |

## 13. インフラスタック推奨

| レイヤ | 技術候補 | 理由 |
|--------|---------|------|
| クラウド基盤 | AWS / Azure / GCP | マルチクラウド対応も検討。DR用に2クラウド利用も可能 |
| コンテナオーケストレーション | Kubernetes (EKS/AKS/GKE) | 自動スケーリング、自己修復 |
| アプリケーションランタイム | Node.js + Python（API層）/ Java/Spring（バッチ処理） | チーム技術選定による |
| データベース | PostgreSQL（OLTP）+ TimescaleDB（時系列/IoTデータ） | ACID保証、パーティショニング対応 |
| キャッシュ | Redis Cluster | 高スループット、クラスタ対応 |
| メッセージキュー | Apache Kafka | イベント駆動アーキテクチャ、大規模メッセージ処理 |
| ロギング | ELK Stack (Elasticsearch + Logstash + Kibana) / OpenSearch | 構造化ログの収集・可視化 |
| 監視 | Prometheus + Grafana + Alertmanager | メトリクス収集、ダッシュボード、アラート通知 |
