# 食品製造業MES - システムアーキテクチャ定義書

## 1. 全体構成概要

### 1.1 アーキテクチャ設計方針

本システムは以下の原則に基づき設計する。

- **マルチファクトリ対応**: 複数工場を論理的に分離しつつ、本社視点での横断管理も可能とする
- **サービス指向アーキテクチャ(SOA)**: ドメイン駆動設計(DDD)に基づくドメインごとのサービス分割
- **イベント駆動型**: Kafkaによる非同期イベントバスでサービス間の疎結合を実現
- **IF連携ファースト**: 既存システムとの統合を最初から考慮した設計
- **コンテナネイティブ**: Docker + Kubernetesでのデプロイ・運用

## 3. データベース設計方針

### 3.1 スキーマ分離

各サービスが独立したスキーマ（またはデータベースインスタンス）を持つ。共通マスタのみ共有スキーマとする。

```
┌─────────────────────────────────────────────┐
│         共用スキーマ (shared_schema)          │
│  - factoryマスタ                            │
│  - production_lineマスタ                    │
│  - productマスタ                            │
│  - materialマスタ                           │
│  - user/roleマスタ                          │
│  - location_hierarchies（工場/倉庫構造）     │
└──────────────┬──────────────────────────────┘
               │ FK参照のみ
   ┌───────────┼───────────┬───────────┐
   ▼           ▼           ▼           ▼
inv_schema   hac_schema  mfg_schema  qlt_schema
```

### 3.2 工場分離キー

全データテーブルに `factory_id` を付与。クエリで自動フィルタリングする。

```sql
-- 全テーブル共通カラム
factory_id      VARCHAR(20) NOT NULL, -- FK: factoryマスタ
line_id         VARCHAR(20),          -- FK: production_lineマスタ（オプション）
tenant_scope    VARCHAR(20),          -- 論理分離用のテナントスコープ

-- 監査カラム（全テーブル共通）
created_at      TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
updated_at      TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
deleted_at      TIMESTAMP WITH TIME ZONE,     -- 論理削除対応
```

### 3.3 TimescaleDB活用（IoTデータ）

```sql
-- センサー記録用ヒップテーブル
CREATE TABLE sensor_records (
    time        TIMESTAMPTZ NOT NULL,
    factory_id  VARCHAR(20) NOT NULL,
    line_id     VARCHAR(20),
    device_id   VARCHAR(50) NOT NULL,
    sensor_type VARCHAR(20) NOT NULL, -- temperature, humidity, pressure, weight...
    value       DOUBLE PRECISION NOT NULL,
    unit        VARCHAR(10) NOT NULL,  -- °C, %, kPa, g...
    status      VARCHAR(10) DEFAULT 'OK', -- OK, WARN, ALERT, OFFLINE
    created_at  TIMESTAMPTZ DEFAULT NOW()
);

SELECT create_hypertable('sensor_records', 'time');
```

## 5. マルチファクトリ対応設計

### 5.1 データ分離方式

- **論理分離**: `factory_id` カラムで全データを区別
- **クエリフィルタリング**: すべてのSELECTにfactory_id条件を自動付与（ミドルウェアレベル）
- **権限制御**: ユーザーは所属工場のみアクセス可能（クロス工場アクセスは管理者のみ）

### 5.2 グローバル管理機能

```
本社ダッシュボード:
├── 全工場の生産実績比較（折れ線グラフ、棒グラフ）
├── 全工場の在庫状況（カテゴリ別ピッキング率）
├── CCP逸脱発生件数/工場別集計
├── 品質検査合格率の横断比較
└── アレルギー混入リスク時のクロス工場トレース
```

### 5.3 工場間データ共有モデル

| データタイプ | 共有範囲 | 説明 |
|------------|---------|------|
| グローバルマスタ | 全工場共通 | 原材料・製品マスタ、品質基準 |
| 工場固有マスタ | 自工場のみ | 工程フロー、CCP条件、保管場所定義 |
| トランザクションデータ | 自工場のみ | ロット履歴、生産記録、検査結果 |
| 監査証跡 | 全工場（管理者） | 全工場の操作ログを一元管理 |

## 6. 主要データフロー

### 6.1 原材料受入 → FIFO在庫登録フロー

```
[サプライヤー]発注書
        │
        ▼
[システム] PO作成 ────→ [ERP]受注同期(IF)
        │
        ▼ (物品受領時)
[倉庫オペレーター] スキャンで検収
        │
        ├────▶ [location-service]: 保管場所割当（温度帯マッチ）
        ├────▶ [inventory-service]: ロット採番 + FIFO登録
        │        ├─ lot_no生成: LOT-{material}-{factory}-{YYYYMMDD}-{seq}
        │        ├─ expiry_date設定（賞味期限優先、または受入日+保存期間）
        │        └─ fifo_order = MAX(fifo_order) + 1
        ├────▶ [audit-service]: 監査ログ記録
        └────▶ [kafka]: inventory.lot.created イベント発行
                         │
                         ▼
              [traceability-service] フォワード起点を登録
              [notification-service] 管理者へ受領完了通知
```

### 6.2 生産指示 → FIFO原材料投入フロー

```
[生産計画] バッチ指示 ────→ [ERP]受注確認(IF)
        │
        ▼
[mfg-service] BOM展開 + 資材引き当て
        │
        ├────▶ [inventory-service]: FIFOソート実行
        │        ├─ expiry_date順にソート
        │        ├─ lot_reservation作成（状態: RESERVED）
        │        └─ 分割対応: 1 LOTから複数LOTへの部分引き当て可能
        │
        ▼ (ラインOPによる投入時)
[オペレーター] バーコードスキャンでLOT確認
        │
        ├────▶ [inventory-service]: RESERVED → CONSUMED
        ├────▶ [mfg-service]: バッチ進捗更新
        ├────▶ [haccp-service]: CCP記録トリガー（必要に応じて）
        └────▶ [kafka]: manufacturing.material.consumed イベント
```

### 6.3 HACCP CCP記録フロー

```
[製造ライン] CCP監視ポイント
        │
        ▼ (リアルタイム)
[ioTゲートウェイ] センサーデータ取得
        │
        ▼
[haccp-service]: CCP記録自動生成・評価
        │
        ├─ 正常値: CCP_RECORD.status = PASS
        ├─ 逸脱範囲内: CCP_RECORD.status = WARN + 通知送信
        └─ 逸脱範囲外: CCP_RECORD.status = FAIL + アラート発令
                         │
                         ▼ (FAIL時)
              [kafka]: haccp.ccp.deviation イベント
                         │
                         ├────▶ [quality-service]: NCR自動起票
                         ├────▶ [notification-service]: 重大逸脱アラート(SMS+メール)
                         ├────▶ [audit-service]: 改ざん防止ハッシュチェーン更新
                         └───▶ [mfg-service]: バッチ停止フラグON（ライン制御連動）
```

### 8.2 HACCP記録の不可変性保証

- CCP記録はCREATE後、UPDATE/DROPを禁止（トリガーで制御）
- ハッシュチェーンで連番改ざんを検知
- 月次SHA-256マッシュ由る整合性チェックを自動化
- 電子署名（電署法準拠）による法的証拠力確保
