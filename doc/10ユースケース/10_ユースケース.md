# 食品製造業MES - ユースケース定義書

## 1. 概要
本ドキュメントは、食品製造業向けMESシステムのユースケースを定義する。
アクター（役割）ごとに分類し、各ユースケースの詳細なフローと前提条件・事後条件を記載する。

---

## 2. アクター一覧

| アクターID | 名称 | 説明 |
|-----------|------|------|
| ACT-001 | 生産マネージャー | 生産計画の策定、指示書の発行、実績確認 |
| ACT-002 | 倉庫担当者 | 原材料・製品の受入、保管、仕分け、出荷 |
| ACT-003 | HACCP責任者 | HACCP計画の管理、CCP監視データの確認、逸脱対応 |
| ACT-004 | 品質検査員 | 原材料・工程内・最終製品の検査実施 |
| ACT-005 | 設備保守担当 | 設備の定期点検・修理記録 |
| ACT-006 | 管理者（システム） | ユーザー管理、マスタ管理、権限設定 |
| ACT-007 | 外部監査人 | HACCP監査・品質監査の実施 |
| ACT-008 | IoTセンサー | 温度・重量・pH等の自動データ収集 |

## 4. 各ユースケースの詳細

### UC-01: 生産指示書の発行

| 項目 | 内容 |
|------|------|
| アクター | ACT-001（生産マネージャー） |
| 前提条件 | BOMが登録されている、在庫に原材料がある |
| 事後条件 | 生産指示書が発行され、原材料がリザーブされる |

**メインフロー:**
```
1. 生産マネージャーは新規生産指示を作成する
2. システムは対象製品を選択
3. システムはBOMを展開し必要原材料を計算
4. システムはFIFO順で原材料ロットを自動選択（リザーブ）
5. 資材不足がある場合は警告を表示
6. 生産マネージャーが内容を確認して発行
7. システムは倉庫に仕分け指示を送信
```

**代替フロー:**
- 4a. FIFO適用の例外: 正当な理由を入力して手動でロットを選択可能（記録される）
- 5a. 資材不足: サプライヤーへの発注を促す警告を表示

---

### UC-02: 生産実績の記録

| 項目 | 内容 |
|------|------|
| アクター | ACT-001（生産マネージャー） / 作業者 |
| 前提条件 | 生産指示書が発行されている |
| 事後条件 | 各工程の実績が記録される |

**メインフロー:**
```
1. 作業者が生産バッチの工程を開始する
2. システムは標準レシピ・手順を表示
3. 各工程で原材料を投入 → ロット番号スキャン入力
4. システムはFIFO順のロットから引き当てているか検証
5. CCP工程では監視データを記録（手動入力またはIoT自動取得）
6. 工程完了時に終了時刻を記録
7. 全工程完了時、完成品Lot IDを自動発行
```

---

### UC-03: バッチ別生産レポート

| 項目 | 内容 |
|------|------|
| アクター | ACT-001（生産マネージャー） |
| 前提条件 | 生産実績が記録されている |
| 事後条件 | レポートが生成・表示される |

**メインフロー:**
```
1. 管理者がバッチIDまたは期間を指定
2. システムは該当バッチの全実績を集計
3. 投入量、完成量、歩留まり、所要時間、CCP記録を表示
4. PDF/CSVでエクスポート可能
```

---

### UC-04: 原材料の受入

| 項目 | 内容 |
|------|------|
| アクター | ACT-002（倉庫担当者） |
| 前提条件 | PO（発注書）が存在する |
| 事後条件 | 在庫が登録され、保管場所が割り当てられる |

**メインフロー:**
```
1. 倉庫担当者がPO番号を入力し受入伝票を作成
2. 物理的に物品を秤量・検収
3. システムはロットIDを発行（LOT-R-YYYYMMDD-XXX）
4. 製造元から有効期限を確認して入力
5. 保管場所を決定（温度帯適合チェックを実施）
6. COA（分析成績書）PDFをアップロード
7. 検収結果（OK/条件OK/NG）を入力
8. システムは在庫テーブルを更新、倉庫位置にマッピング
```

**代替フロー:**
- 5a. 温度帯不一致: エラーを表示し、適切な保管場所への移動を促す
- 7a. NGの場合: NCR（不適合品報告）を自動発行し隔離区域へ移動指示

---

### UC-05: ロットの分割・投入

| 項目 | 内容 |
|------|------|
| アクター | ACT-002（倉庫担当者） |
| 前提条件 | 原材料ロットに在庫がある、生産指示が発行されている |
| 事後条件 | 元ロットの一部数量が分割され、新ロットが作成される |

**メインフロー:**
```
1. システムは生産指示に基づき必要な原材料を提示（FIFO順）
2. 倉庫担当者は対象ロットの在庫を確認
3. 取り出し数量を入力 → ロット分割を実行
4. システムは:
   a) 新しい子Lot IDを発行
   b) 親ロットの在庫数を減算
   c) 子ロットを「生産使用中」状態に変更
5. バーコードスキャンで保管場所を確認し記録
```

---

### UC-06: FIFO在庫出し入れ

| 項目 | 内容 |
|------|------|
| アクター | ACT-002（倉庫担当者） |
| 前提条件 | 在庫に物品が存在する |
| 事後条件 | 在庫数量が減少し、出庫履歴が記録される |

**メインフロー:**
```
1. システムは対象製品のFIFO順リストを提示
2. 倉庫担当者は指示されたロットを確認・取り出し
3. バーコードでLot IDを検証 → FIFO順と一致するかチェック
4. 不一致の場合は警告（理由は必須入力）
5. 出庫伝票を作成し在庫を更新
```

---

### UC-07: CCP監視データの記録

| 項目 | 内容 |
|------|------|
| アクター | ACT-003（HACCP責任者） / ACT-008（IoTセンサー） |
| 前提条件 | CCPが設定されている工程で製造が進行中 |
| 事後条件 | CCP監視データが記録され、クリティカルリミットとの比較結果が表示される |

**メインフロー:**
```
1. システムはCCP監視をすべき工程・タイミングを表示（アラート）
2. 手動入力の場合: HACCP責任者が実測値を入力
3. IoT連携の場合: センサーが自動データを送信
4. システムはクリティカルリミットと比較
   - 正常範囲内 → 「OK」として記録
   - 逸脱 → 即時警告（画面+通知）+ 是正手順を表示
5. 是正措置が必要な場合はUC-08へ連携
```

---

### UC-08: CCP逸脱時の是正措置

| 項目 | 内容 |
|------|------|
| アクター | ACT-003（HACCP責任者） |
| 前提条件 | CCP監視データがクリティカルリミットを逸脱 |
| 事後条件 | 是正処置が記録され、影響範囲が評価される |

**メインフロー:**
```
1. UC-07でCCP逸脱を検知 → 警告発出
2. HACCP責任者は画面の是正手順に従う
3. 実施した是正処置を入力（対象ロット数、処分方法等）
4. システムは影響を受けた他のLotを自動検索
5. アレルギー混入の場合は即時全関係者へ通知
6. 記録は監査証跡に保存
```

---

### UC-09: HACCP記録のエクスポート

| 項目 | 内容 |
|------|------|
| アクター | ACT-003（HACCP責任者） / ACT-007（外部監査人） |
| 前提条件 | HACCP関連の記録が蓄積されている |
| 事後条件 | PDF/CSV形式で監査用レポートが生成される |

**メインフロー:**
```
1. レポート対象期間とタイプを選択
2. システムは該当期間の全HACCP記録を集約
3. HACCP計画書、CCP実績、逸脱記録等を一括エクスポート
4. 監査証跡（改竄防止ハッシュ）付きのPDFを生成
```

### UC-10: 原材料受入検査の実施

| 項目 | 内容 |
|------|------|
| アクター | ACT-004（品質検査員） |
| 前提条件 | 原材料が受入済み |
| 事後条件 | 検査結果が記録され、合格なら「検収済」状態に、不合格ならNCR発行 |

**メインフロー:**
```
1. システムは受検対象のロットと検査項目を表示
2. 品質検査員は各項目を測定・入力（重量、温度、外観等）
3. システムは規格値と比較して判定（OK/NG）
4. 全項目がOK → ロット状態を「検収済」に変更
5. NGがある場合 → NCR発行 + 隔離指示
```

---

### UC-11: 最終製品検査と出荷判断

| 項目 | 内容 |
|------|------|
| アクター | ACT-004（品質検査員） |
| 前提条件 | 製造が完了し、完成品が倉庫入库済み |
| 事後条件 | 出荷承認またはリジェクト |

**メインフロー:**
```
1. システムは最終製品検査の指示書を表示
2. 検査員はサンプリングを実施（JIS/ASQ基準に基づき自動計算）
3. 各項目を検査・入力
4. CCP/CQIデータも併せて確認
5. システムが出荷判断を提案（全データOKなら自動OK）
6. 管理者が最終承認（必要に応じて手動判定）
7. 承認後 → 出荷指示書を作成可能になる
```

---

### UC-12: 設備定期点検の実施

| 項目 | 内容 |
|------|------|
| アクター | ACT-005（設備保守担当） |
| 前提条件 | 設備に定期点検スケジュールが設定されている |
| 事後条件 | 点検結果が記録される |

**メインフロー:**
```
1. システムは到期予定の点検をアラートとして表示
2. 設備保守担当者が現地で点検を実施
3. 結果（正常/異常、修理内容）を入力
4. 異常の場合は修理依頼を自動発行
5. CCP関連機器の場合、校正結果も記録（UC-07の信頼性に関わる）
```

| 事後条件 | ユーザーまたはマスタデータが更新される |

**メインフロー:**
```

2. 新規ユーザー/マスタ項目の登録フォームを表示
3. データを入力・保存

### UC-14: 監査用レポートの生成

| 項目 | 内容 |
|------|------|
| アクター | ACT-007（外部監査人） |
| 前提条件 | システムに記録が蓄積されている |
| 事後条件 | 監査対応レポート一式がエクスポート可能になる |

**メインフロー:**
```
1. 監査人が監査タイプを選択（HACCP監査/品質監査等）
2. 監査期間を指定
3. システムは関連する全レポートを自動生成:
   - HACCP計画書と実施記録
   - CCP監視データ
   - プリ・レquisites記録
   - 機器校正履歴
   - NCR/CAR追跡記録
4. 改竄防止ハッシュ付きPDFを一括出力
5. 監査人が確認・署名（電子署名）
```

### UC-15: IoTセンサーデータ自動取得

| 項目 | 内容 |
|------|------|
| アクター | ACT-008（IoTセンサー） - システム自動 |
| 前提条件 | センサーが接続され、CCP設定がされている |
| 事後条件 | CCP監視データとして自動記録される |

**メインフロー:**
```

4. 正常 → データベースに保存、異常 → UC-07へ連携
5. 全データは監査証跡に記録
```

## 6. カテゴリ別シナリオ（製品カテゴリ固有のユースケースフロー）

### 6.1 レトルト食品 - バッチ殺菌フロー (UC-03 + UC-HACCP-01)

```
アクター: 生産ラインオペレーター → HACCP責任者 → 倉庫担当者

1. [UC-03] 生産指示書受領
   - SKU: RC-001（レトルトカレー）バッチBATCH-20260514-005を受理
   
2. [UC-02] 原材料引き当て（FIFO）
   - LOT-R-0501(鶏肉,20kg) + LOT-R-0428(鶏肉,10kg) = 30kg必要 → 両方使用
   - LOT-S-0510(ソース,25kg) = 25kg必要 → 完全使用
   
3. [UC-03] 工程進行: 殺菌工程到達
   - オートクレーブ温度: 121.2℃（CCP-1基準: ≥121℃）→ OK
   - F値計算: 3.15（CCP-1基準: ≥3.0）→ OK
   
4. [UC-HACCP-01] CCP逸脱時の処理（例外的シナリオ）
   - 仮にF値が2.8の場合: 即時アラート → HACCP責任者に通知
   - 是正措置: バッチ再殺菌または廃棄判断
   - 記録: 逸脱理由・是正内容・担当者を全て記録
   
5. [UC-04] 完成品入库
   - LOT-FG-20260514-005を常温倉庫(ZONE-ROOM)に配置
